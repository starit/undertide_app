# Theming guide

This app uses **CSS custom properties** (HSL triplets, space-separated) for all product colors, **Tailwind** to map those tokens to utilities, and **next-themes** to persist a theme choice on `<html class="‚Ä¶">`.

The single source of truth for palette values is [`app/globals.css`](../app/globals.css).

---

## How it fits together

| Layer | Role |
|--------|------|
| `:root` | Default **Light** theme ‚Äî same variable names as other themes |
| `.dark` | **Dark** theme ‚Äî overrides every token used by the UI |
| `.sepia` | **Sepia** (warm editorial) ‚Äî optional third palette |
| [`tailwind.config.ts`](../tailwind.config.ts) | Maps `background`, `foreground`, `brand`, `destructive`, `shadow-panel`, `bg-grid`, etc. to `hsl(var(--token))` |
| [`components/theme-provider.tsx`](../components/theme-provider.tsx) | Registers allowed `themes` for `next-themes` |
| [`components/theme-switcher.tsx`](../components/theme-switcher.tsx) | Lets users pick a saved theme |

Tailwind is set to `darkMode: ["class"]` so any `dark:` utilities track the `.dark` class on the root element.

---

## Variable contract (what every theme must define)

When adding a theme, duplicate the **full set** of variables from `:root` and adjust values. Missing keys fall through to the wrong palette.

**Core UI**

- `--background`, `--foreground`
- `--card`, `--card-foreground`
- `--popover`, `--popover-foreground`
- `--primary`, `--primary-hover`, `--primary-foreground`
- `--secondary`, `--secondary-foreground`
- `--muted`, `--muted-foreground`
- `--accent`, `--accent-foreground`
- `--destructive`, `--destructive-foreground`
- `--border`, `--input`, `--ring`
- `--brand`, `--brand-foreground`, `--brand-muted`
- `--info`, `--info-foreground`
- `--warning`, `--warning-foreground`

**Page chrome & effects**

- `--page-radial-accent`, `--page-linear-highlight` ‚Äî body background gradients
- `--selection-highlight` ‚Äî usually `var(--primary)` for branded text selection

**Markdown code fences / syntax**

- `--code-background`, `--code-border`, `--code-header-background`, `--code-header-foreground`, `--code-foreground`
- `--syntax-comment`, `--syntax-keyword`, `--syntax-string`, `--syntax-function`, `--syntax-number`

**Elevation & avatars**

- `--shadow-color`, `--grid-line` ‚Äî panel shadow and hero grid (often `var(--foreground)`)
- `--avatar-gradient-sat`, `--avatar-gradient-light-a`, `--avatar-gradient-sat-accent`, `--avatar-gradient-light-b` ‚Äî deterministic author avatar gradients

**Browser chrome**

- Set `color-scheme: light` or `dark` on the theme block for native scrollbars/form controls.

## Add a new named theme (checklist)

1. **Pick a class name** (lowercase, no spaces), e.g. `ocean`.  
   Avoid reserved words that clash with Tailwind or HTML.

2. **In `app/globals.css`**, after `.sepia` (or after `.dark`), add:

   ```css
   .ocean {
     color-scheme: light; /* or dark */
     /* copy every variable from :root and edit */
   }
   ```

3. **Register the theme** in `components/theme-provider.tsx`:

   ```tsx
   themes={["light", "dark", "sepia", "ocean", "system"]}
   ```

4. **Expose it in the UI** in `components/theme-switcher.tsx` (label + icon).

5. **Smoke-test**: home hero (grid + shadow), cards, `<select>` popovers, proposal markdown (links, quotes, **code block** + syntax), flagged warning strip, and author avatars.

6. **Optional:** If the PWA chrome should match, update [`app/manifest.ts`](../app/manifest.ts) `theme_color` / `background_color` ‚Äî they do not read CSS variables.

## Light vs system vs custom themes

- **light** ‚Äî uses `:root` tokens only (no extra class required beyond what `next-themes` applies).
- **dark** ‚Äî `next-themes` adds `class="dark"`; `.dark { ‚Ä¶ }` supplies tokens.
- **system** ‚Äî resolves to light or dark from OS preference; still uses `:root` or `.dark`.
- **Custom** (e.g. **sepia**) ‚Äî `class="sepia"` on `<html>`; overrides must live in `.sepia { ‚Ä¶ }`.

Only one theme class is active at a time.

---

## Tips

- Keep **HSL** in the same shape as existing tokens: `H S% L%` without wrapping `hsl()`.
- For **contrast**, check primary/foreground pairs and destructive on both `background` and `card`.
- **Code blocks** are intentionally a separate mini-palette so they stay readable; tune them per theme rather than reusing `card` colors blindly.
