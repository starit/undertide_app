# Theming guide

This app uses **CSS custom properties** (HSL triplets, space-separated) for all product colors, **Tailwind** to map those tokens to utilities, and **next-themes** to persist a theme choice on `<html class="…">`.

The single source of truth for palette values is [`app/globals.css`](../app/globals.css).

---

## How it fits together

| Layer | Role |
|--------|------|
| `:root` | Default **Light** theme — same variable names as other themes |
| `.dark` | **Dark** theme — overrides every token used by the UI |
| `.sepia` | **Sepia** (warm editorial) — optional third palette |
| [`tailwind.config.ts`](../tailwind.config.ts) | Maps `background`, `foreground`, `brand`, `destructive`, `shadow-panel`, `bg-grid`, etc. to `hsl(var(--token))` |
| [`components/theme-provider.tsx`](../components/theme-provider.tsx) | Registers allowed `themes` for `next-themes` |
| [`components/theme-switcher.tsx`](../components/theme-switcher.tsx) | Lets users pick a saved theme |

Tailwind is set to `darkMode: ["class"]` so any `dark:` utilities track the `.dark` class on the root element.

---

## Variable contract (what every theme must define)

When adding a theme, duplicate the **full set** of variables from `:root` and adjust values. Missing keys fall through to the wrong palette.

**Core UI**

- `--background`, `--foreground`
- `--card`, `--card-foreground`
- `--popover`, `--popover-foreground`
- `--primary`, `--primary-hover`, `--primary-foreground`
- `--secondary`, `--secondary-foreground`
- `--muted`, `--muted-foreground`
- `--accent`, `--accent-foreground`
- `--destructive`, `--destructive-foreground`
- `--border`, `--input`, `--ring`
- `--brand`, `--brand-foreground`, `--brand-muted`
- `--info`, `--info-foreground`
- `--warning`, `--warning-foreground`

**Page chrome & effects**

- `--page-radial-accent`, `--page-linear-highlight` — body background gradients
- `--selection-highlight` — usually `var(--primary)` for branded text selection

**Markdown code fences / syntax**

- `--code-background`, `--code-border`, `--code-header-background`, `--code-header-foreground`, `--code-foreground`
- `--syntax-comment`, `--syntax-keyword`, `--syntax-string`, `--syntax-function`, `--syntax-number`

**Elevation & avatars**

- `--shadow-color`, `--grid-line` — panel shadow and hero grid (often `var(--foreground)`)
- `--avatar-gradient-sat`, `--avatar-gradient-light-a`, `--avatar-gradient-sat-accent`, `--avatar-gradient-light-b` — deterministic author avatar gradients

**Browser chrome**

- Set `color-scheme: light` or `dark` on the theme block for native scrollbars/form controls.

## Add a new named theme (checklist)

1. **Pick a class name** (lowercase, no spaces), e.g. `ocean`.  
   Avoid reserved words that clash with Tailwind or HTML.

2. **In `app/globals.css`**, after `.sepia` (or after `.dark`), add:

   ```css
   .ocean {
     color-scheme: light; /* or dark */
     /* copy every variable from :root and edit */
   }
   ```

3. **Register the theme** in `components/theme-provider.tsx`:

   ```tsx
   themes={["light", "dark", "sepia", "ocean", "system"]}
   ```

4. **Expose it in the UI** in `components/theme-switcher.tsx` (label + icon).

5. **Smoke-test**: home hero (grid + shadow), cards, `<select>` popovers, proposal markdown (links, quotes, **code block** + syntax), flagged warning strip, and author avatars.

6. **Optional:** If the PWA chrome should match, update [`app/manifest.ts`](../app/manifest.ts) `theme_color` / `background_color` — they do not read CSS variables.

## Light vs system vs custom themes

- **light** — uses `:root` tokens only (no extra class required beyond what `next-themes` applies).
- **dark** — `next-themes` adds `class="dark"`; `.dark { … }` supplies tokens.
- **system** — resolves to light or dark from OS preference; still uses `:root` or `.dark`.
- **Custom** (e.g. **sepia**) — `class="sepia"` on `<html>`; overrides must live in `.sepia { … }`.

Only one theme class is active at a time.

## themed.js (`--themed-*`)

The app loads [**themed.js**](https://github.com/starit/themed.js) (`@themed.js/core`, `@themed.js/react`) for **`--themed-*`** tokens and AI theme generation. **Default UI** still comes from `globals.css` + `next-themes` until the user generates an AI theme — then **HSL variables on `<html>`** are overridden to match the generated palette (Tailwind reads the same `--background`, `--primary`, etc.). **`next-themes`** still controls the `light` / `dark` / `sepia` / `system` class on `<html>`; changing that dropdown clears AI HSL overrides and restores the static palette for the chosen preset.

| Piece | Role |
|--------|------|
| `components/theme-provider.tsx` | Wraps `next-themes` with `ThemedJsThemeProvider` from `@themed.js/react`. |
| `components/themed-js-next-sync.tsx` | Maps `resolvedTheme` → themed.js builtin: `light` / `dark` / `system` → `light` / `dark`; **sepia** → **rose** (closest warm builtin). Clears AI HSL overrides when the preset changes. |
| `components/themed-js-ssr-style.tsx` | Renders `ThemeScript` inside a **client** boundary (do not import `@themed.js/react` directly from the RSC `app/layout.tsx` — it breaks the server bundle). Injects `<style id="themed-js-styles">` for first paint. |
| `components/ai-hsl-theme-rehydrate.tsx` | After load, if `sessionStorage` has an AI theme, re-imports it and reapplies HSL overrides on `<html>`. |
| `components/themed-font-bridge.tsx` | Syncs `tokens.typography.fontFamily` from the active themed.js theme onto `--font-sans` / `--font-serif` / `--font-mono` for Tailwind. |
| `lib/theme-font-bridge.ts` | Helpers to apply or clear those font CSS variables on `<html>`. |
| Storage | themed.js uses `storage: { type: 'none' }` so it does not fight `next-themes` localStorage; sync keeps variables aligned after hydration. |

**Natural-language token themes:** the header **AI style** box calls `POST /api/themes/generate` (DeepSeek, `DEEPSEEK_API_KEY`). The response is applied to **themed.js** (`--themed-*`) and, via `lib/ai-theme-to-app-hsl.ts`, to **the same HSL custom properties** Tailwind uses (`--background`, `--primary`, code-fence tokens, etc.) on `document.documentElement`. The theme JSON is kept in `sessionStorage` so a refresh can rehydrate (`AiHslThemeRehydrate`). Choosing a different preset in the **Theme** dropdown clears AI overrides and restores `globals.css` palettes for that preset.

Hooks: `useTheme` / `useAITheme` from `@themed.js/react` (inside `ThemeProvider`). Full API: [themed.js repo](https://github.com/starit/themed.js).

### AI theme → app HSL (readability)

`lib/ai-theme-to-app-hsl.ts` maps a small set of hex **color** tokens onto the full app HSL contract; contrast is **not guaranteed**. Practical improvements:

1. **Post-process WCAG** — After building triplets, check pairs (e.g. primary vs primary-foreground, text vs background) with themed.js contrast helpers or a small HSL nudge loop until AA or a max iteration.
2. **Stricter LLM contract** — Tune the generation path (prompt / `customSchema`) so the model must return palettes that already satisfy contrast rules, reducing heuristic load.
3. **Clamp to safe ranges** — Derive each role from bounded H/S/L regions anchored to the active builtin preset (light/dark) so AI can only “wobble” within a safe corridor.
4. **Hybrid mapping** — Apply AI only to accent/brand tokens; keep core `--background` / `--foreground` from a hand-tuned preset until checks pass.
5. **User-facing “Boost contrast”** — One click applies deterministic lightness/saturation adjustments without a full regeneration.

## Tips

- Keep **HSL** in the same shape as existing tokens: `H S% L%` without wrapping `hsl()`.
- For **contrast**, check primary/foreground pairs and destructive on both `background` and `card`.
- **Code blocks** are intentionally a separate mini-palette so they stay readable; tune them per theme rather than reusing `card` colors blindly.
