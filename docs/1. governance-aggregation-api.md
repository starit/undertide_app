# Governance Aggregation API And Data Flow

Status: source-specific APIs exist; aggregate protocol/proposal APIs are still planned.

UnderTide is already a governance product, so public routes should not use a `/api/governance/*` prefix. The route model is:

- source-specific APIs under `/api/<source>/*`
- aggregate product APIs at top-level resources: `/api/sources`, `/api/sync`, `/api/protocols`, `/api/proposals`

Use source objects/source refs as the only intermediate source-object concept.

## Current Route Families

### Source-Specific APIs

Source routes expose one upstream using that source's native concepts.

```txt
GET /api/snapshot/sync
GET /api/snapshot/spaces
GET /api/snapshot/spaces/[slug]
GET /api/snapshot/spaces/[slug]/proposals
GET /api/snapshot/proposals
GET /api/snapshot/proposals/[id]
GET /api/snapshot/proposals/[id]/translations

GET /api/tally/sync
GET /api/tally/organizations
GET /api/tally/organizations/[idOrSlug]
GET /api/tally/organizations/[idOrSlug]/proposals
GET /api/tally/proposals
GET /api/tally/proposals/[id]
```

Rules:

- Use plural collection names: `proposals`, not `proposal`.
- Keep source-specific shapes close to the upstream source model.
- Do not force Tally organizations to be called spaces in `/api/tally/*`.
- Do not expose raw upstream JSON by default.

### Legacy Compatibility APIs

These routes remain Snapshot-compatible during migration:

```txt
GET /api/spaces
GET /api/spaces/[slug]
GET /api/spaces/[slug]/proposals
GET /api/proposals
GET /api/proposals/[id]
GET /api/proposals/[id]/translations
GET /api/sync/snapshot
GET /api/sync/tally
```

Migration target:

- Current frontend client fetches should use `/api/snapshot/*`.
- `/api/proposals` should become the aggregate proposal API only after all Snapshot-compatible consumers are migrated.
- Keep legacy routes until external callers have a replacement path.

Recommended deprecation headers when old aliases are retired:

```txt
Deprecation: true
Link: </api/snapshot/proposals>; rel="successor-version"
```

## Source Identity

Every aggregate source-backed entity must include:

```ts
type GovernanceSource = "snapshot" | "tally";

type SourceScopedId = `${GovernanceSource}:${string}`;
```

Rules:

- `source` is the upstream platform.
- `sourceId` is the native upstream ID.
- `uid` is the stable aggregate ID, for example `snapshot:<proposalId>` or `tally:<proposalId>`.
- Parse `uid` by splitting on the first colon only.
- Source-specific detail routes accept native source IDs.
- Aggregate detail routes accept `uid`.

Source descriptors live in `lib/governance/sources.ts`.

## Product Model

Use three product concepts:

```ts
type GovernanceProtocol = {
  id: string;
  slug: string;
  name: string;
  aliases: string[];
  categories: string[];
  website: string | null;
  avatar: string | null;
  sourceRefs: GovernanceProtocolSourceRef[];
};

type GovernanceProtocolSourceRef = {
  protocolId: string;
  source: GovernanceSource;
  sourceId: string;
  sourceKind: "snapshot-space" | "tally-organization" | "tally-governor";
  sourceSlug: string | null;
  sourceName: string | null;
  isPrimary: boolean;
  confidence: "manual" | "high" | "medium" | "low";
};

type GovernanceProposal = {
  uid: SourceScopedId;
  source: GovernanceSource;
  sourceId: string;
  protocolId: string | null;
  sourceObject: {
    uid: SourceScopedId;
    source: GovernanceSource;
    sourceId: string;
    sourceKind: GovernanceProtocolSourceRef["sourceKind"];
    slug: string | null;
    name: string | null;
  };
};
```

Practical meaning:

- A protocol is the canonical product entity, for example `uniswap`.
- A protocol source ref links that protocol to a Snapshot space, Tally organization, or Tally governor.
- A proposal belongs to one source object and may optionally resolve to a canonical protocol through source refs.
- If no protocol mapping exists yet, `protocolId` is `null`; the proposal should still be returned.

## Storage Model

Existing source tables:

- Snapshot: `snapshot_spaces`, `snapshot_proposals`, `snapshot_space_members`
- Tally: `tally_organizations`, `tally_proposals`

Source tables should keep:

- normalized columns needed for efficient filtering and sorting
- `raw jsonb not null` with the complete upstream response
- `synced_at`
- `created_at` and `updated_at`

Planned aggregate mapping tables:

```txt
governance_protocols

governance_protocol_sources

- source text not null
- source_id text not null
```

## Internal Sync Pipeline

### Stage 1: Source Ingest

Snapshot:

- Script: `scripts/sync-snapshot.ts`
- Writes: `snapshot_spaces`, `snapshot_space_members`, `snapshot_proposals`
- Sync state entity types: `spaces`, `proposals`
- Current public read APIs: `/api/snapshot/*`

Tally:

- Script: `scripts/sync-tally.ts`
- Writes: `tally_organizations`, `tally_proposals`
- Sync state entity types: `tally:organizations`, `tally:proposals`
- Current public read APIs: `/api/tally/*`

Rules:

- Source ingest does not decide canonical protocol identity.
- Source ingest must preserve upstream `raw`.
- Source ingest should only store enough normalized columns for source-specific filtering and aggregate query pushdown.

### Stage 2: Protocol Source Linking

This is a planned job/script.

Suggested script:

```txt
scripts/link-governance-protocols.ts
```

Inputs:

- manually curated seed map for high-value protocols
- `snapshot_spaces`
- `tally_organizations`
- optional future Tally governors table if needed

Outputs:

- `governance_protocols`
- `governance_protocol_sources`

Matching strategy:

1. Manual seeds win.
2. Exact normalized slug match is high confidence.
3. Exact normalized name match is medium confidence.
4. Heuristic matches require review and should be stored as low confidence or skipped.

Do not silently merge protocols on weak similarity. Wrong canonical merges are worse than missing mappings.

Seed example:

```ts
{
  protocolId: "uniswap",
  name: "Uniswap",
  sources: [
    { source: "snapshot", sourceId: "uniswapgovernance.eth", sourceKind: "snapshot-space" },
    { source: "tally", sourceId: "2206072040315953936", sourceKind: "tally-organization" }
  ]
}
```

### Stage 3: Aggregate Query Normalization

Aggregate APIs should normalize at query time first:

```txt
source tables -> source query helpers -> normalizers -> merge/sort/page -> API response
```

Suggested modules:

```txt
lib/governance/types.ts
lib/governance/normalizers.ts
lib/governance/repository.ts
lib/governance/cursor.ts
```

Normalizers:

```ts
normalizeSnapshotSpaceToSourceObject(row)
normalizeTallyOrganizationToSourceObject(row)
normalizeSnapshotProposal(row, context)
normalizeTallyProposal(row, context)
normalizeProposalStatus(source, sourceStatus)
```

## Aggregate Proposal Query Algorithm

For `source=all`:

1. Normalize and validate query params.
2. Resolve `protocolId` to source refs when present.
3. Build one SQL query per source with pushdown filters:
   - Snapshot: `space_id`, `state`, `title/body q`, time order, limit
   - Tally: `organization_id`, `status`, `chain_id`, `title/description q`, time order, limit
4. Fetch `limit + 1` rows per source.
5. Normalize each source row to `GovernanceProposalListItem`.
6. Merge in memory using the requested sort.
7. Return `limit` rows and emit cursor from the final returned item.

For `source=snapshot` or `source=tally`, use only that source query and still return aggregate response shape.

```ts
type ListResponse<T> = {

```

Detail routes may return:

```json
{ "data": {} }
```

Errors should return:

```json

```

## Cursor Design

Use cursor pagination, not offsets, for aggregate APIs.

Cursor payload before base64url encoding:

```json
{
  "v": 1,
  "sort": "time",
  "queryHash": "sha256-of-normalized-query",
  "lastValue": "2026-05-01T10:00:00.000Z",
  "lastUid": "tally:123"
}
```

Rules:

- Include cursor version.
- Include query hash so cursors cannot be reused with different filters.
- Include primary sort value and `uid` tie-breaker.
- Cursors are opaque to clients.
- Reject malformed cursors with `400`.

## Status Normalization

Expose both coarse and detailed status:

```ts
type GovernanceStatusGroup = "Upcoming" | "Active" | "Closed" | "Executed";

type GovernanceOutcome = "passed" | "failed" | "unknown" | null;
```

Snapshot:

| Source state | `statusGroup` | `status` | `outcome` |
| --- | --- | --- | --- |
| `pending` | `Upcoming` | `Pending` | `unknown` |
| `active` | `Active` | `Active` | `unknown` |
| `closed` | `Closed` | `Closed` | `unknown` |
| anything else | `Executed` | source state | `unknown` |

Tally:

| Source state | `statusGroup` | `status` | `outcome` |
| --- | --- | --- | --- |
| `PENDING` | `Upcoming` | `Pending` | `unknown` |
| `ACTIVE` | `Active` | `Active` | `unknown` |
| `SUCCEEDED` | `Closed` | `Succeeded` | `passed` |
| `QUEUED` | `Closed` | `Queued` | `passed` |
| `EXECUTABLE` | `Closed` | `Executable` | `passed` |
| `EXECUTED` | `Executed` | `Executed` | `passed` |
| `DEFEATED` | `Closed` | `Defeated` | `failed` |
| `EXPIRED` | `Closed` | `Expired` | `failed` |
| `CANCELED` | `Closed` | `Canceled` | `unknown` |
| anything else | `Closed` | source state | `unknown` |

## Translation Compatibility

Current `proposal_translations` rows are Snapshot-scoped by `proposal_id`. Do not reuse that table for Tally without adding source identity.

Preferred future table:

```txt
governance_proposal_translations
- source
- source_id
- locale
- title
- body
- summary
- translated_by
- created_at
- updated_at
```

Until that exists:

- `locale` overlay in aggregate proposal routes should apply only to Snapshot records.
- Tally records should return source-language content.
- `translatedOnly=true` should exclude non-Snapshot records unless source-scoped translations are implemented.
