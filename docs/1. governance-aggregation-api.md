# Governance API Route Design

Status: route direction updated; source-specific routes are being introduced before the aggregate API replaces legacy paths.

UnderTide is a governance product, so the public API does not need a `/api/governance/*` prefix. The preferred route model is:

- source-specific APIs under `/api/<source>/*`
- aggregate product APIs at top-level resources like `/api/proposals`, `/api/venues`, and `/api/protocols`

The existing Snapshot-compatible `/api/spaces` and `/api/proposals` routes should remain stable during migration. New source-specific clients should use `/api/snapshot/*` or `/api/tally/*`.

## Goals

- Make source-specific and aggregate APIs visibly different.
- Preserve source identity so callers can trace every aggregate item back to Snapshot, Tally, or another upstream.
- Avoid lossy normalization in the ingest layer. Raw source tables remain durable source data.
- Keep filtering, sorting, pagination, and joins in repository/API code first; do not add UI-only aggregation.
- Let future sources plug in through source namespaces and normalizers rather than requiring a product API rewrite.

## Route Families

### Source-Specific APIs

Source-specific routes expose one upstream source using that source's native concepts.

```txt
GET /api/snapshot/sync
GET /api/snapshot/spaces
GET /api/snapshot/spaces/[slug]
GET /api/snapshot/spaces/[slug]/proposals
GET /api/snapshot/proposals
GET /api/snapshot/proposals/[id]
GET /api/snapshot/proposals/[id]/translations

GET /api/tally/sync
GET /api/tally/organizations
GET /api/tally/organizations/[idOrSlug]
GET /api/tally/organizations/[idOrSlug]/proposals
GET /api/tally/proposals
GET /api/tally/proposals/[id]
```

Rules:

- Use plural collection names: `proposals`, not `proposal`.
- Keep source-specific shapes close to the source's data model.
- Do not force Tally organizations to be called spaces in `/api/tally/*`.
- Do not expose raw upstream JSON by default. Add an explicit debug/admin path later if needed.

### Legacy Compatibility APIs

These routes currently remain Snapshot-compatible:

```txt
GET /api/spaces
GET /api/spaces/[slug]
GET /api/spaces/[slug]/proposals
GET /api/proposals
GET /api/proposals/[id]
GET /api/proposals/[id]/translations
GET /api/sync/snapshot
GET /api/sync/tally
```

Migration target:

- `/api/spaces` can eventually become an alias of `/api/venues`, or remain a Snapshot compatibility endpoint.
- `/api/proposals` should eventually become the aggregate proposal API after frontend and external callers are ready.
- During migration, prefer deprecation headers over breaking response shapes.

Recommended deprecation headers when old aliases are retired:

```txt
Deprecation: true
Link: </api/snapshot/proposals>; rel="successor-version"
```

### Aggregate APIs

Aggregate APIs expose normalized product concepts across all sources.

```txt
GET /api/sources
GET /api/sync

GET /api/protocols
GET /api/protocols/[id]

GET /api/venues
GET /api/venues/[uid]
GET /api/venues/[uid]/proposals

GET /api/proposals
GET /api/proposals/[uid]
```

These aggregate routes are the long-term API for product UI, agents, and external consumers.

## Source Identity

Every aggregate entity must include:

```ts
type GovernanceSource = "snapshot" | "tally";

type SourceScopedId = `${GovernanceSource}:${string}`;
```

Rules:

- `source` is the upstream platform.
- `sourceId` is the native upstream ID.
- `uid` is the stable aggregate ID, for example `snapshot:<proposalId>` or `tally:<proposalId>`.
- Parse `uid` by splitting on the first colon only.
- Public aggregate detail routes should accept `uid`, not bare upstream IDs.
- Source-specific detail routes should accept native source IDs.

Source descriptors live in `lib/governance/sources.ts`.

## Product Model

Avoid treating every source object as the same thing. Web3 governance needs at least three concepts:

```ts
type GovernanceProtocol = {
  id: string;
  name: string;
  aliases: string[];
  sources: GovernanceSource[];
};

type GovernanceVenue = {
  uid: SourceScopedId;
  source: GovernanceSource;
  sourceId: string;
  protocolId: string | null;
  kind: "snapshot-space" | "tally-organization" | "tally-governor";
};

type GovernanceProposal = {
  uid: SourceScopedId;
  source: GovernanceSource;
  sourceId: string;
  venueUid: SourceScopedId;
  protocolId: string | null;
};
```

Practical meaning:

- A protocol is the canonical product entity, for example `uniswap`.
- A venue is where governance happens, for example a Snapshot space or Tally organization/governor.
- A proposal belongs to one source venue.
- The first aggregate implementation may leave `protocolId` null until canonical matching exists.

## Storage Model

Keep source tables separate:

- Snapshot: `snapshot_spaces`, `snapshot_proposals`, `snapshot_space_members`
- Tally: `tally_organizations`, `tally_proposals`

Both source families should store:

- normalized columns needed for efficient filtering and sorting
- `raw jsonb not null` with the complete upstream response
- `synced_at`
- `created_at` and `updated_at`

The first aggregation phase should normalize from these source tables in `lib/governance/repository.ts` or `lib/repository.ts`. Add materialized views or aggregate tables only after query volume and indexes justify them.

## Response Envelope

List routes should return:

```ts
type ListResponse<T> = {
  data: T[];
  pageInfo: {
    nextCursor: string | null;
    hasNextPage: boolean;
  };
  meta: {
    sources: Array<{
      source: GovernanceSource;
      syncedAt: string | null;
      partial: boolean;
      error: string | null;
    }>;
  };
};
```

Detail routes may return the existing envelope:

```json
{ "data": {} }
```

Errors should return:

```json
{ "error": "message" }
```

## Aggregate Venues

`GET /api/venues`

Parameters:

| Param | Type | Description |
| --- | --- | --- |
| `q` | `string` | Search by venue or protocol name. |
| `source` | `"snapshot" \| "tally" \| "all"` | Source filter. Default `all`. |
| `protocolId` | `string` | Canonical protocol filter. |
| `kind` | `"snapshot-space" \| "tally-organization" \| "tally-governor"` | Venue kind filter. |
| `verified` | `"true" \| "false"` | Source-normalized verification filter where available. |
| `hasActiveProposals` | `"true" \| "false"` | Filter venues with active proposals. |
| `sort` | `"activity" \| "followers" \| "proposals" \| "updated"` | Sort order. Default `activity`. |
| `limit` | `number` | Page size. Default `50`, max `200`. |
| `cursor` | `string` | Opaque cursor for stable pagination. |

Response item:

```ts
type GovernanceVenueListItem = {
  uid: SourceScopedId;
  source: GovernanceSource;
  sourceId: string;
  kind: "snapshot-space" | "tally-organization" | "tally-governor";
  protocolId: string | null;
  slug: string;
  name: string;
  summary: string;
  avatar: string | null;
  verified: boolean | null;
  categories: string[];
  followers: number | null;
  proposalsCount: number;
  activeProposalsCount: number;
  website: string | null;
  forum: string | null;
  profileUrl: string;
  syncedAt: string;
};
```

## Aggregate Proposals

`GET /api/proposals`

Once migrated, this route should be the multi-source proposal API. Until migration is complete, use `/api/snapshot/proposals` for Snapshot-compatible behavior.

Parameters:

| Param | Type | Description |
| --- | --- | --- |
| `q` | `string` | Search proposal title, body summary, or venue/protocol name. |
| `source` | `"snapshot" \| "tally" \| "all"` | Source filter. Default `all`. |
| `protocolId` | `string` | Canonical protocol filter. |
| `venueUid` | `string` | Aggregate venue ID, for example `snapshot:aave.eth`. |
| `venueSlug` | `string` | Source venue slug filter where supported. |
| `statusGroup` | `"Upcoming" \| "Active" \| "Closed" \| "Executed" \| "All"` | Coarse UI lifecycle status. |
| `status` | `string` | Normalized detailed status. |
| `chainId` | `string` | Chain filter, mostly useful for Tally/on-chain proposals. |
| `sort` | `"time" \| "heat" \| "votes" \| "endingSoon"` | Sort order. Default `time`. |
| `locale` | `string` | Optional translation overlay. Initially Snapshot-only. |
| `translatedOnly` | `"true" \| "false"` | Effective only where translations exist. |
| `limit` | `number` | Page size. Default `50`, max `200`. |
| `cursor` | `string` | Opaque cursor for stable pagination. |

Response item:

```ts
type GovernanceProposalListItem = {
  uid: SourceScopedId;
  source: GovernanceSource;
  sourceId: string;

  title: string;
  body: string | null;
  summary: string;

  protocol: {
    id: string | null;
    name: string | null;
  };

  venue: {
    uid: SourceScopedId;
    source: GovernanceSource;
    sourceId: string;
    kind: "snapshot-space" | "tally-organization" | "tally-governor";
    slug: string;
    name: string;
    avatar: string | null;
  };

  statusGroup: "Upcoming" | "Active" | "Closed" | "Executed";
  status: string;
  sourceStatus: string;
  outcome: "passed" | "failed" | "unknown" | null;

  author: string | null;
  publishedAt: string | null;
  startsAt: string | null;
  endsAt: string | null;

  votesCount: number | null;
  quorum: string | null;
  heat: number;

  discussionUrl: string | null;
  proposalUrl: string;
  sourceUrl: string;

  labels: string[];
  chainId: string | null;
  syncedAt: string;
};
```

## Status Normalization

Expose both coarse and detailed status:

```ts
type GovernanceStatusGroup = "Upcoming" | "Active" | "Closed" | "Executed";

type GovernanceOutcome = "passed" | "failed" | "unknown" | null;
```

Snapshot:

| Source state | `statusGroup` | `status` | `outcome` |
| --- | --- | --- | --- |
| `pending` | `Upcoming` | `Pending` | `unknown` |
| `active` | `Active` | `Active` | `unknown` |
| `closed` | `Closed` | `Closed` | `unknown` |
| anything else | `Executed` | source state | `unknown` |

Tally:

| Source state | `statusGroup` | `status` | `outcome` |
| --- | --- | --- | --- |
| `PENDING` | `Upcoming` | `Pending` | `unknown` |
| `ACTIVE` | `Active` | `Active` | `unknown` |
| `SUCCEEDED` | `Closed` | `Succeeded` | `passed` |
| `QUEUED` | `Closed` | `Queued` | `passed` |
| `EXECUTABLE` | `Closed` | `Executable` | `passed` |
| `EXECUTED` | `Executed` | `Executed` | `passed` |
| `DEFEATED` | `Closed` | `Defeated` | `failed` |
| `EXPIRED` | `Closed` | `Expired` | `failed` |
| `CANCELED` | `Closed` | `Canceled` | `unknown` |
| anything else | `Closed` | source state | `unknown` |

Put this mapping in `lib/governance/normalizers.ts` when the aggregate API is implemented.

## Pagination

Use cursor pagination, not offsets, for aggregate APIs.

Cursor payload before base64url encoding:

```json
{
  "sort": "time",
  "lastValue": "2026-05-01T10:00:00.000Z",
  "lastUid": "tally:123"
}
```

Rules:

- Include the primary sort value and `uid` tie-breaker.
- Cursors are opaque to clients.
- Cursor validation should reject unknown sort modes and malformed dates.
- When merging multiple sources in memory, fetch `limit + 1` from each relevant source, merge, then emit the next cursor from the final returned item.

## Translation Compatibility

Current `proposal_translations` rows are Snapshot-scoped by `proposal_id`. Do not reuse that table for Tally without adding source identity.

Preferred future table:

```txt
governance_proposal_translations
- source
- source_id
- locale
- title
- body
- summary
- translated_by
- created_at
- updated_at
```

Until that exists:

- `locale` overlay in aggregate proposal routes should apply only to Snapshot records.
- Tally records should return source-language content.
- `translatedOnly=true` should exclude non-Snapshot records unless source-scoped translations are implemented.

## Implementation Order

1. Done: introduce `/api/snapshot/*` aliases for current Snapshot-compatible behavior.
2. Done: introduce `/api/tally/*` read routes backed by Tally raw tables.
3. Done: keep legacy `/api/spaces` and `/api/proposals` stable during migration.
4. Done: add `/api/sources` and `/api/sync` discovery/status endpoints.
5. In progress: migrate frontend consumers from legacy routes to source-specific or aggregate routes.
6. Next: add `lib/governance/types.ts` and `lib/governance/normalizers.ts`.
7. Next: add `/api/venues` and aggregate `/api/proposals`.
8. Later: add deprecation headers to legacy aliases only after the frontend no longer depends on them.
9. Later: revisit materialized views or aggregate tables only after measuring query cost.
